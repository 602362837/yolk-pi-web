## Reusable knowledge
# Summary

修复了 WorkTree 归档/删除后，对应项目空间仍残留在 Project Registry/Sidebar 中的问题，并覆盖了 CLI/直接删目录等非 UI 场景。

# Reusable knowledge

- 对 worktree space 采用**软清理**而非硬删除：标记 `archived: true, missing: true`，保留历史 session header 与用户自定义空间元数据，便于未来同路径 worktree 复用。
- 后端统一使用 `archiveWorktreeSpacesByPaths` 做清理：
  - 主匹配键为 `pathKey`
  - `displayPath` / `realPath` 仅作同一 space 的精确兜底
  - 只处理 `space.kind === "worktree"`
  - 支持 archive/delete 时传入多个路径别名统一命中
- 非 UI 场景需要**被动同步**：新增 `syncMissingWorktreeSpaces` 做 missing-only 检测，不依赖 git 命令；适合在 `GET /api/projects?sync=missing` 与 worktree refresh 中触发。
- 被动同步若实际归档了 missing worktree space，需要同步 `invalidateAllowedRootsCache()`，避免文件 API 短时间继续授权已失效路径。
- `GET /api/projects?sync=missing` 应先执行同步，再返回最新 registry，避免“同步已发生但响应仍是旧数据”。
- 前端 `SessionSidebar` 在归档/删除成功后不能只处理 session fallback，必须刷新 projects，并按 **main → 同项目其他 active/non-missing space → fallbackCwd → null** 进行选中空间回退。
- 当前剩余风险仅为非阻塞项：Project Registry 仍是读-改-写原子落盘，但无专门进程内写锁；未来若高频并发 archive/refresh/sync，可能仍有最后写入覆盖旧写入的老风险。
