| `components/SessionSidebar.tsx` | Project Registry-driven project/space tree sidebar. It loads projects from `/api/projects` instead of synthesizing them from sessions, and lazy-loads **active** linked sessions for the selected project space via `GET /api/projects/:projectId/spaces/:spaceId/sessions` (response `{ sessions, legacyUnassigned, studioChildrenByParentSessionId }` only — no `archivedCounts`, no `sessions-archive/` scan). The sidebar does **not** load, expand, or restore archived session lists: there is no “已归档 (N)” block, no `archivedSessions` state, and no client call to `/api/sessions/archived` or unarchive. Empty space state is active-only (`No sessions found in this space`). Single/batch/`archive-all` write actions remain (POST archive / archive-all); success only refreshes the active list via `loadSessions(false)`, and archive-all confirmation counts the current active `filteredSessions.length`. It hides YPI Studio child audit sessions from ordinary root history by default while showing them as nested Studio audit rows under visible parents (main titles via `displayTitleForSession()` / shared `studioChildSessionTitle()` so bound implementer/checker children show `subtaskId · subtaskTitle`, while architect/improver children keep `member · taskTitle`; badge/detail/tooltip still carry member, status, and run short id), shows exact-cwd legacy unassigned sessions separately, keeps WorkTree actions available for worktree spaces, and includes the integrated file explorer with a resizable session/explorer split. **Project/space selection** has been replaced from an inline dropdown to a compact top switch button (`选择项目空间` when empty, or `ProjectName / SpaceName` when selected). Clicking the button opens `ProjectSpaceSwitchDialog` for layered (left project list → right space list) browsing, search, and add-project flows inside a viewport-level modal dialog. The `SessionSidebar` owns all project/space state and callbacks (registration from manual path, OS directory picker, Git clone, default directory) and passes them into the dialog as props; the dialog is purely a UI shell that calls back to the sidebar's existing `upsertProjectAndSelectMainSpace()`, `handleDirectoryPicker()`, `commitCustomPath()`, `handleGitCloneSubmit()`, etc. The sidebar header displays current project name / space subtitle / WorkTree badge; **Current workspace menu** is a single shared content unit (`CurrentWorkspaceMenuContent` / `currentWorkspaceMenu` state): the project-space switch button right-click opens fixed `context` mode with viewport clamp, and the adjacent ⋯ button opens `anchored` mode under the trigger — both render the same project/space metadata, pin, archive-all-sessions, archive-space, and archive-project actions with the same callbacks. When the selected space is a WorkTree, both triggers append the shared trailing “归档 WorkTree…” / “删除 WorkTree…” items and reuse `openWorktreeAction` / existing confirm dialogs; ordinary spaces never show those items. Escape, outside click, action execution, and selection changes close the menu; no empty menu when no workspace is selected. Left-click on the switch button still opens `ProjectSpaceSwitchDialog`. Dialog-internal arbitrary project/space right-click menus (`ProjectSpaceContextMenuState`) remain separate and are not the current-workspace menu. WorkTree archive/delete for non-selected dialog rows remains available by right-clicking a WorkTree row inside the dialog. Add-project forms (manual path, Git clone) are rendered inside the dialog with mutually exclusive toggles; empty registry onboarding renders directly in the dialog with large primary buttons for `Add project folder…` / `Add project path…` / `Add project from Git…` / `Use default directory`. The dialog closes on successful registration/clone and selects the returned main space; on failure the current project is unchanged. Main spaces without a custom display name render as `主空间` while keeping `spaceId: "main"`; the selected space header subtitle shows `空间：...` plus the current Git branch or WorkTree branch/base status, appends path-missing state when needed, and keeps full path/status details in the hover/focus tooltip. WorkTree badges remain compact (`WT` plus branch) with a tooltip for branch and base/unknown state. Session, Studio child, hover-action, and delete-confirmation rows keep title/metadata content on single non-wrapping lines and ellipsize long text at narrow sidebar widths. **WorkTree archive/delete flow:** `confirmWorktreeAction()` optimistically merges the API response's `archivedSpaces` into local project state so the worktree space disappears immediately, then refreshes projects in the background. If the currently-selected space was the archived/deleted one, fallback preference is: (1) `main` space of the same project, (2) any other active non-missing space in that project, (3) API-provided `fallbackCwd`, (4) null. Deleted session ids are forwarded via `onSessionDeleted`. The sidebar **centralizes project and space context menu state** through `ProjectSpaceContextMenuState` (targeting a specific project or project+space pair). Context menu callbacks (`handleDialogProjectContextMenu`, `handleDialogSpaceContextMenu`) bridge dialog right-click events into the sidebar state so that edit/star/archive/WorkTree-delete actions operate on the right-clicked target, not just the currently selected workspace. **Space drag-reorder** is handled by `handleReorderSpaces` which optimistically updates local state and calls `PATCH /api/projects/[projectId]/spaces`, rolling back to pre-drag project state on failure. **Top workspace menu and metadata dialog** labels have been unified to "星标/取消星标" for both projects and spaces; the metadata dialog distinguishes the two with contextual hints ("星标项目优先排序" for projects vs "仅用于标记，不影响空间排序" for spaces). |

### Action tags, SVG icon-flow, and control radius

**Opt-in icon stroke-flow** is a reusable visual capability: independent stroke action icons can show a dashed gradient overlay that animates on the host’s motion policy. Tech feel is limited to **inline SVG stroke flow**; button borders/backgrounds stay static state surfaces. Chrome such as `.tech-action-tag` is **optional** — hosts may keep their existing button shape and only add `data-icon-flow` + `ActionFlowIcon`.

#### Three-step opt-in (required)

1. Host element sets motion policy via `{...iconFlowAttrs("interactive" | "ambient" | "off")}` (or equivalent `data-icon-flow`).
2. Replace stroke geometry with `<ActionFlowIcon …>{path/line/rect/…}</ActionFlowIcon>` (same viewBox/geometry; do not invent a second flow SVG stack).
3. Do **not** add global `button` / unscoped hover animation rules. Without `data-icon-flow`, overlay stays hidden even if `ActionFlowIcon` is present.

#### Modes

| Mode | When it flows | Allowed hosts |
| --- | --- | --- |
| `interactive` | `:hover`, `:focus-visible`, and open/active via `.is-active` / `aria-expanded="true"` (not bare `aria-pressed` alone — pressed toggles such as theme stay static at rest and only flow on hover/focus) | Default for new whitelist replacements |
| `ambient` | Continuous low-frequency while enabled | **Only** sidebar utility strip (`.sidebar-utility-actions`: Models / Usage / Skills / Settings). Stagger delays are scoped to that container. Do not spread ambient to top bar, forms, or panels. |
| `off` | Never (overlay hidden) | Disabled, loading/spinning, or explicit static states |
| *(no attr)* | Never | Default everywhere |

#### Maximum reasonable replacement (whitelist strategy)

Replace **independent** stroke icon actions / toolbar actions when geometry maps cleanly to stroke `ActionFlowIcon`. Do **not** require pill tag chrome. Batches already applied in production code:

| Batch | Hosts (representative) |
| --- | --- |
| B0 | `AppShell` sidebar utility (ambient) + top-bar actions (interactive) |
| B1 | `ChatInput` attach/send/steer/follow-up/compact(idle)/toggles; `BrowserShareControl` bind pill; `MessageView` copy/edit/fork-style message actions |
| B2 | `SessionSidebar` new session / worktree / refresh / workspace menu; `AppShell` explorer space-info refresh |
| B3 | `FileViewer` add-to-chat toolbar; Usage refresh (`ChatGptUsagePanel`, `UsageProviderModelTable`); `ModelsConfig` toolbar refresh/test/show-key/copy-key; `SkillsConfig` add skill |
| B4 | `TerminalPanel` collapse, new local, open SSH, maximize (interactive). Close-tab / close-dock / terminate remain static |

#### Blacklist (must stay static — never add `data-icon-flow`)

| Category | Examples |
| --- | --- |
| Destructive | Session-row Delete/Archive, Browser Share 解绑, reject command, Stop Agent solid square, compacting solid fill |
| Close / remove chips | TabBar close, modal close, attachment thumbnail remove ×, Terminal close tab / close dock & kill processes |
| Dense list / tree / table rows | Session row rename/delete/expand forks, file-tree row actions, model account Disable/Delete rows |
| Form decoration / segments | `SelectDropdown` chevron, segmented control cells, checkbox |
| Stats chip body | `SessionStatsChips` triggers |
| Dual animation / spin | `GitPanel` refresh (existing spin) — do not stack stroke-flow |
| Uncontrolled geometry | `FileIcons`, external images, font icons, pure fill decoration |
| Drag / resize | Resize handles without a clean stroke action |

#### Chat primary Send contrast (hard gate)

`.chat-input-primary-action--send` (and peer steer/follow-up focus rules) must **not** bleach the whole control to white with a light/white icon on focus/hover/active. Production CSS keeps hover/focus/active on **accent background + white label/icon**, focus via outline/ring/`box-shadow`, and pale `--icon-flow-a/b/c` stops so the flow overlay stays readable on accent. Do not reintroduce “white button + light icon” prototypes for these controls.

#### Tokens / classes

| Token / class / attr | Role |
| --- | --- |
| `--control-radius` (8px), `--control-radius-sm` (6px), `--control-radius-pill` (999px) | Ordinary control radius baseline + pill / compact variants. |
| `--control-motion-fast`, `--icon-flow-ambient`, `--icon-flow-interactive`, `--icon-flow-active`, `--icon-flow-dash`, `--icon-flow-travel`, `--icon-flow-a/b/c` | Static transition + dash motion timing/colors (theme-overridden in `html.dark`). |
| `:where(button)` | Low-specificity ordinary-button 8px radius. Explicit inline/class radius on circular, pill, segmented, and special controls continues to win. **Do not** add `button { border-radius: … !important }`. |
| `.tech-action-tag`, `.tech-action-tag--icon` | **Optional** pill action tag chrome (height 28px, static 1px border/surface, focus ring, active/disabled). Icon-only is 28×28. Not a prerequisite for icon-flow. |
| `.sidebar-utility-actions`, `.sidebar-utility-tag` | Sidebar bottom grid: 30px tags; default 4 columns; `.is-narrow` or `.sidebar-container[data-sidebar-width="220"]` → 2×2. AppShell sets these when `sidebarWidth <= 220`. Ambient stagger is scoped here. |
| `.app-top-actions`, `.app-top-action-tag` | Top-bar cluster; tags sit in the existing 36px bar; mobile still hides `.app-top-label` and keeps 28×28 icon tags + horizontal scroll. |
| `.action-flow-icon`, `__base`, `__overlay` | Base `currentColor` stroke + gradient dashed overlay; only overlay dash offset animates. |
| `[data-icon-flow="ambient\|interactive\|off"]` | Host-agnostic motion policy (see Modes). CSS requires a descendant `.action-flow-icon__overlay`. |
| `.chat-input-primary-action`, `--send` / `--steer` / `--followup` | Chat primary action contrast + focus ring (see hard gate above). |
| `.tech-action-tag__badge` | Subagents/Git badges above SVG (`z-index: 2`); mobile inset 1–2px. |

#### Motion / a11y contract

- Flow only on visible SVG stroke/path geometry (base + overlay). No continuous border, background, or pseudo-element sweep.
- Host-agnostic selectors in `app/globals.css`: `[data-icon-flow=…] … .action-flow-icon__overlay`. No `button { animation… }` / unscoped `button:hover` flow rules.
- `:disabled` / `.is-disabled` / `data-icon-flow="off"` hide overlay and cancel animation; base stroke remains.
- `@media (prefers-reduced-motion: reduce)` hides all icon-flow overlays and disables tag color transitions; static active/open surfaces remain.
- Overlay is decorative (`aria-hidden`, `pointer-events: none`); accessible names stay on the native `button` via label / `aria-label` / `title`.
- Per-instance gradient ids (`useId` sanitized) so multiple icons on one screen do not clash.

#### Reuse / anti-patterns

- Prefer `iconFlowAttrs` + `ActionFlowIcon` for new independent stroke actions on the whitelist.
- Do **not** invent `FlowIconButton` or a second stroke-flow stack; do not restore edge-flow/conic border glow for these controls.
- Do **not** “maximize range” by breaking the blacklist or forcing ambient outside the sidebar utility strip.
- Do not change `onClick`, panel open state, dropdown anchors, badges, or business handlers when migrating icons.

#### Rollback

1. Revert host migrations (restore inline SVG, remove `data-icon-flow` / `iconFlowAttrs`).
2. Revert or narrow `app/globals.css` icon-flow / chat-primary-action blocks.
3. Remove `components/iconFlow.ts` and this documentation section if fully abandoning the capability.
4. No data migration; batches are file-local and can roll back independently.

### AppShell hydration-safe layout persistence

`components/AppShell.tsx` keeps four localStorage-backed layout values on hydration-safe external stores (`createPersistentLayoutStore` + `useSyncExternalStore`), not `useState(getInitial*)` with unconditional persist effects:

| State | Storage key(s) | Server snapshot | Client notes |
| --- | --- | --- | --- |
| `sidebarWidth` | `pi-web-sidebar-width` | `260` | Clamp 220–520 on read/write. |
| `rightPanelWidth` | `pi-web:right-panel-width` | `300` (`RIGHT_PANEL_MIN_WIDTH`) | Clamp to viewport (min 300, max ~65%); missing/invalid falls back to ~42% viewport after hydration. |
| `explorerHeight` | `pi-web-preview-explorer-height`; legacy `pi-web-sidebar-explorer-height` (one-shot migrate, also in `watchKeys`) | `null` | Min height 120; `null` keeps flex default; write of `null` is a no-op (does not clear stored preference). | `explorerOpen` | `pi-web-preview-explorer-open` | `true` | Affects conditional FileExplorer subtree; missing key defaults open (`stored !== "false"`). |
