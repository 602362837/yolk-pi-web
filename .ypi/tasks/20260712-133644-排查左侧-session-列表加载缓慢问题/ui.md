# UI 门禁判断

## 结论

当前方案 **不触发 UI 原型门禁**。

原因：规划仅优化请求合并、取消、服务端扫描/缓存/索引和内部计时；保留现有侧栏内容、排序、加载状态、错误状态、刷新按钮和空间切换行为，不新增用户可见信息结构或交互。

## 边界

以下任一选择会触发门禁，届时必须派发 `ui-designer`，基于现有 `SessionSidebar` 产出任务目录内 `.html` 原型并等待用户审批：

- 空间切换时保留旧列表并显示后台刷新状态。
- 新增 skeleton、阶段提示、超时提示、重试入口或性能诊断信息。
- 改变刷新按钮反馈、禁用状态或请求中的交互。
- 引入分页、无限滚动、虚拟列表并改变用户浏览方式。

当前无 HTML 原型，也不得在上述交互变更出现时直接进入实现。
