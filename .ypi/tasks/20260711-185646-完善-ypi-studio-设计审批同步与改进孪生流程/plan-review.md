# 第三版：改进流程设计审批书

## 先看改进流程

“改进流程”是用户在**主任务验收**时提出问题后，在该主任务下面创建的一条可追踪处理链。它不是新顶层任务，也不是普通 child task。

用户确认问题要处理后，main 创建一个带编号的改进项，例如 `IMP-001 修复窄屏布局`。改进师先把反馈收敛成范围、风险和计划；有 UI 变化时 UI 设计员交付 HTML 原型；然后实现员实现、检查员检查，main 再请求用户验收该改进项。任何未解决项都会让主任务显示“等待改进完成”，不能完成或归档。所有项都验收完成后，主任务只回到 review，main 必须再次请求用户验收主任务。

这让用户能清楚知道：问题在哪里、谁在处理、为什么被卡住、下一步由谁做，以及最终仍由哪一个主任务交付。

## 本次要批准的体验

- 浮窗只显示主任务，但明确显示 `待处理改进 N 项`、具体阻塞和下一步。
- 主任务详情新增 `改进流程` Tab，按 `IMP-001` 等编号列出标题、状态、负责人、计划/验收和下一步；展开后可看自己的产物、进度、运行、审批与验收。
- Settings -> Studio 新增 `improver / 改进师`，排在架构师后、UI 设计员前，可独立配置模型和思考强度。
- 主任务验收无问题时直接完成；有问题时先确认创建改进项，全部处理后回到 review 并二次验收。
- 计划审批、原型和审批书仍同步更新，但它们是支撑改进计划可信的配套机制，不取代改进流程本身。

## 需要确认的决定

1. 是否新增默认 `improver` 成员？推荐：是。
2. failed/cancelled/rejected 改进是否继续阻塞，直到用户明确“接受不处理”？推荐：是。
3. 是否允许已批准范围内的明确小修继承主计划批准？推荐：允许；任何 UI、范围、行为或风险变化仍单独批准。
