# UI - 第三版改进流程原型门禁

## 门禁

本需求改变主任务验收、任务详情、浮窗、Settings 成员行和用户可见阻塞信息，**必须有 HTML 原型和用户批准后才能实现**。本版原型：[studio-improvement-flow-v3-prototype.html](studio-improvement-flow-v3-prototype.html)。

## 页面和文案

### 任务浮窗

浮窗仍只代表主任务。进入等待状态时显示：

- 状态 badge：`等待改进完成`。
- 摘要：`待处理改进 2 项`；若没有活跃项则不显示该行。
- 阻塞：`阻塞：IMP-002 等待你的计划批准`，失败时为 `阻塞：IMP-003 已取消，等待“接受不处理”`。
- 下一步：`下一步：在绑定聊天中批准 IMP-002 的计划` 或 `下一步：复核全部改进并再次验收主任务`。
- 详情入口按钮：`查看改进`。不提供“完成”“批准”“接受不处理”等面板直接写状态按钮。

### 任务详情

在现有任务详情新增 `改进流程` Tab；Overview 同时出现简短“改进状态”区块。

- 空态标题：`还没有改进项`；说明：`在主任务验收时确认用户反馈后，改进项会显示在这里。`
- 列表标题：`改进项（2）`。
- 每张列表卡显示：`IMP-002`、标题、状态（例如 `等待计划批准`）、负责人（`改进师`）、`计划：待批准`、`验收：未请求`、更新时间和一行阻塞/下一步。
- 点开后显示五个页签：`概览`、`产物`、`执行进度`、`成员运行`、`审批与验收`。
- 概览包含用户反馈、范围/非目标、风险、当前负责人和下一步；产物显示该项自己的 brief/prd/ui/design/implement/checks/plan-review/HTML；执行进度显示子任务和检查结论；成员运行只显示有界摘要；审批与验收显示当前 revision、是否继承批准、用户请求和最终验收记录。
- 主任务处于 `waiting_for_improvements` 时，完成与归档控件禁用，旁边显示：`仍有 2 个改进项未解决。`

### Settings -> Studio

“默认成员”按以下固定顺序显示：`architect / 架构师`、`improver / 改进师`、`ui-designer / UI 设计员`、`implementer / 实现员`、`checker / 检查员`。改进师行与其他成员完全一致：模型选择器、思考强度选择器，以及来自 Members Tab 的“修改模型”聚焦。

说明文案：`改进师在用户验收反馈后负责把问题收敛成可批准的改进计划；不直接实现代码。`

## 交互和状态

| 场景 | 看到什么 | 可做什么 |
| --- | --- | --- |
| 用户验收无问题 | `验收中` 与 `接受主任务` 请求 | 在绑定聊天明确接受 |
| 用户反馈待确认 | `已收到反馈，等待确认创建改进项` | main 在聊天中确认范围 |
| 改进分析 | `IMP-001 分析中`、负责人“改进师” | 查看摘要和成员运行 |
| 等待计划批准 | revision、审批书、HTML 原型（需要时） | 在绑定聊天明确批准或提出修改 |
| 实现/检查 | 当前子项、检查结论、下一步 | 仅查看；成员由 main 调度 |
| 用户验收改进 | `请验收 IMP-001` | 在绑定聊天接受或提出新的问题 |
| 失败/取消 | 清楚的失败原因和 `等待用户接受不处理` | 用户在聊天明确处置；不能伪装为完成 |
| 全部解决 | `改进已完成，主任务需要再次验收` | 打开主任务 review |

## 可访问性和实现约束

复用 `YpiStudioPanel`、`YpiStudioSessionWidget`、现有 Tabs/Badge/Notice 和 task-local Markdown/HTML 预览。状态不能只靠颜色；通知使用 `aria-live`；详情入口可聚焦；窄屏先显示数量、阻塞和下一步。HTML 预览继续使用任务目录安全 resolver 和 CSP sandbox。
