## Summary
本次修复确立了 YPI Studio 设计确认的硬门禁：任务进入 awaiting_approval 时写入 approvalGate 并清除旧 grant；只有后续同一稳定 session context 中用户明确批准（如“确认/批准/开始实现/approve/go ahead”）后写入 approvalGrant，awaiting_approval -> implementing 才允许，override 不能绕过。prompt/extension 也明确要求设计完成后停在 awaiting_approval 并向用户确认。前端同步方面，Studio 面板改为当前 tab 优先与懒/后台加载，后台刷新保留已有任务内容并仅显示轻量刷新状态，AppShell 仅在 Studio 面板打开时推进面板 refreshKey。任务浮窗实时性通过稳定 context key（优先 pi_<sessionId>）和 Studio task 工具结果 task id/key 的去抖重查实现。验证包含 lint、tsc、checker review 与用户手工验证。

## Reusable knowledge
# Summary

修复 YPI Studio 三类问题：设计确认被跳过、Studio 面板工作中持续刷新影响阅读、创建任务后 session 浮窗不实时出现。核心变更是把“设计后必须等待用户确认”从 prompt 软约束升级为代码硬门禁，并同步优化前端刷新与 session-task 绑定策略。

# Reusable knowledge

- 对工作流中的用户审批边，不能只依赖 `reason` 或 prompt；应持久化 approval gate/grant：进入等待确认状态时写 gate 并清旧 grant，只有后续用户输入明确批准后才写 grant。
- `override` 不应绕过安全/确认类 transition，尤其是 `awaiting_approval -> implementing`。
- 批准文本识别应保守：明确批准词才放行，否定/修改类文本必须排除，避免 “do not proceed” 之类误判。
- Web session 与 Studio task 关联应优先使用稳定 context key，如 `pi_<sessionId>`；`pi_process_*` 不适合作为高置信 UI 浮窗证据。
- 长轮询/后台刷新 UI 时应拆分 initial loading 与 refreshing：已有数据时保留内容，只显示轻量刷新提示，避免用户阅读被全屏 loading 打断。
- 工具结果中如果包含 task id/key，应触发前端去抖重查 session task projection，避免用户必须整页刷新才能看到浮窗。
