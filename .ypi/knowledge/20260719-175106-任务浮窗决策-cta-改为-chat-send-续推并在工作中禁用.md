# 任务浮窗决策 CTA → Chat Send 续推并在工作中禁用

## Summary

解决浮窗 CTA 旁路问题：点批准后状态变了但 Chat 不干活、隐式 `inner.prompt` 模型错位、agent 工作时仍可点。

**Hybrid B**：
1. 门禁仍先 `PATCH`（contextId / expectedRevision / grant / plan-review·UI 证据）。
2. 需主会话干活的动作 PATCH 成功后 `onComposeSend` → 当前 Chat `handleSend` 固定引导词（transcript 可见、模型一致）。
3. `agentRunning` 或写操作 busy 时禁用决策与双验收 CTA。
4. Send 失败不回滚已落库状态；partial toast。

动作矩阵：`approve_plan` / `request_plan_changes` / `approve_improvement_plan` = PATCH+Chat Send；`start_user_acceptance` 与改进/主结果验收 = 仅 PATCH。Server 删除 `request_plan_changes` 成功路径的 `bestEffortContinueAfterWidgetRequestPlanChanges` 主路径调用，保留 helper 作回滚。视觉硬约束：不按示意 HTML 改浮窗布局/样式。

实现：`lib/ypi-studio-widget-continue.ts`、ChatWindow `onComposeSendReady`、AppShell 转发、`YpiStudioSessionWidget` busy+续推+toast、route 去双发、docs/tests。Checker Pass（修 AppShell 假成功 throw、revisionTo）。

## Reusable knowledge

- **双控制面是根因**：浮窗 silent PATCH + server `inner.prompt` 与 Chat 模型/SSE/忙碌互斥脱节；续推应走用户消息路径。
- **Hybrid B 优于纯引导或纯 server wake**：原子性靠 PATCH；可见性/模型靠 Chat Send；不要软化审批门禁。
- **接线**：`ChatWindow.handleSend` → `onComposeSendReady` → AppShell ref → widget；unmount 清空；`onComposeSend` 无句柄时 **throw**，避免 toast 假成功。
- **pure helper**：`ypiStudioWidgetActionNeedsChatContinue` + `buildYpiStudioWidgetChatContinuePrompt`；仅三 continue kind；无 HTML/URL；勿 import rpc-manager。
- **busy**：`agentRunning || acceptingInFlight`；confirm 后读 `agentRunningRef` 二次检查；预览可仍可点。
- **server 双发**：Chat 为主路径后，route 勿再 best-effort wake 同一动作。
- **视觉**：生产组件 + 现网 CSS 唯一权威；任务 HTML 仅交互 checklist。
- **结果验收 ≠ 编排续推**：主/改进结果验收只写库，不自动 Chat Send（用户体感“点完成 Chat 没事”是预期）。
