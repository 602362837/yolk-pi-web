# PRD - 第三版：主任务下的改进流程

## 目标

让用户在主任务验收提出的问题有可见、可执行、可验收的闭环，同时不把主任务拆散。用户看到的是“改进项”，成员执行的是“改进流程”，系统仍只保留一个顶层主任务。

## 范围内

- 在主任务验收期创建、跟踪和关闭一个或多个改进项。
- 改进师、UI 设计员、实现员、检查员、main 和用户的协作闭环。
- 浮窗、任务详情、Settings 成员模型策略和通知的改进流程视图。
- 与改进计划相关的设计、原型、审批书同步。

## 范围外

- 改进项不进入顶层任务列表，不可独立绑定聊天、归档或完成。
- 不自动理解任意聊天文字为改进项；main 必须先取得用户明确创建确认。
- 不把历史任务批量迁移或覆盖自定义 workflow/member 文件。

## 用户故事与验收

| ID | 用户需要 | 验收标准 |
| --- | --- | --- |
| R1 | 在主任务验收提出问题后看到问题被接住 | main 确认后创建带稳定 `imp_` id、标题、反馈摘要、状态和负责人的改进项；主任务改为“等待改进完成”。 |
| R2 | 知道改进项不是另一张任务 | 顶层 Tasks 列表和 session pointer 只显示主任务；改进项只在主任务详情、浮窗摘要和主任务目录中出现。 |
| R3 | 看懂现在卡在哪里 | 浮窗显示“待处理改进 2 项”、首个阻塞原因和下一步；详情列表展示每项状态、负责人、审批和验收结果。 |
| R4 | 让反馈被分析而不是直接改代码 | `improver` 先输出范围、非目标、风险、计划与是否需要 UI/批准；信息不清楚时标记“等待澄清”。 |
| R5 | 让成员各做各的事 | ui-designer 仅在 UI 变化时交付 HTML；implementer 只实现已准备的改进子项；checker 给出通过或返工；main 请求用户验收。 |
| R6 | 不让未解决问题被静默关掉 | failed、cancelled、rejected、blocked 都保持未解决并阻止主任务完成；只有用户明确“接受不处理”才能关闭该项并留下理由。 |
| R7 | 让主任务仍有最终验收 | 全部改进项为 accepted 或 accepted_not_doing 后，主任务只回到 `review`；main 必须再次请求用户接受主任务才可 completed。 |
| R8 | 为改进师设置合适模型 | Settings -> Studio 的默认成员表新增 `improver / 改进师`，排在 architect 后、ui-designer 前；可单独设置模型和思考强度，并沿用既有优先级。 |
| R9 | 修改计划时不出现材料不同步 | 每次改进计划修改同步更新 brief/prd/ui/design/implement/checks/plan-review 和 HTML（需要时），旧批准失效，用户只批准最新材料。 |

## 主流程

```text
checking -> review -> user_acceptance
                         |-- 用户接受 -> completed
                         `-- 用户提出问题并确认创建改进项
                                      -> waiting_for_improvements
                                      -> 全部改进验收完成
                                      -> review -> user_acceptance -> completed
```

## 产品决定待确认

- 默认新增 `improver`：推荐是。
- 未处理的失败/取消/拒绝是否持续阻塞：推荐是，只有用户明确“接受不处理”才解除。
- 批准继承：推荐仅限已批准范围内的明确小修；UI、行为范围或风险变化一律独立批准。
