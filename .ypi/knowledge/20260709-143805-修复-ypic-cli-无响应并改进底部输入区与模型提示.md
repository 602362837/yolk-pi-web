# 修复 ypic CLI 无响应并改进底部输入区与模型提示

## Summary

修复了 ypic CLI 的三个核心问题：/model 命令无响应、普通输入静默无反馈、启动提示与输入区体验差。通过 6 个子任务完成：UI 原型审批门禁 → 无响应路径诊断 → /model 命令实现 → SSE/发送可靠性 → TTY 底部固定输入区 → 文档与测试。Checker 阶段额外发现并修复了 PlainFrame close handler 和 /model thinking running 阻断两个边界问题。最终 lint/tsc/53 tests 全部通过。

## Reusable Knowledge

### 1. CLI 双 Frame 渲染架构（TerminalFrame / PlainFrame）
终端 CLI 的用户可见交互改造应采用双 Frame 抽象：TTY 环境使用 `TerminalFrame`（ANSI alternate screen + 固定底栏 + 状态点 + 右侧模型/thinking），非 TTY / `NO_COLOR` / `YPIC_PLAIN=1` 使用 `PlainFrame`（readline 驱动，无 ANSI 污染）。工厂函数 `createFrame(rl)` 根据 `process.stdout.isTTY && process.stdin.isTTY && !NO_COLOR && !YPIC_PLAIN` 选择。关键：PlainFrame 的 `start()` 需接受 `onClose` 回调参数，避免非 TTY positional 模式被 `readline` 的 `close` 事件提前退出。

### 2. SSE Connected Gate 与发送可见状态
CLI 客户端的 `connectSse()` 必须暴露 `connected` promise；首条消息发送前 `await Promise.race([sse.connected, timeout(10s)])`，超时给出明确警告但不阻塞。发送 prompt 前立即 `frame.writeLine("Sending…")` + `frame.setStatusDot("busy")`，POST/preflight/SSE error 统一走 `frame.setStatusDot("error")` + 明确错误消息。这解决了用户"发出去什么都没发生"的核心体验问题。

### 3. SDK Runner `runtime_lost` 恢复策略
当 Studio subagent run 因 `runtime_lost` 失败但底层 pi session JSONL 文件仍在写入时，代码可能已完成但未产出 handoff。恢复步骤：(1) 检查 JSONL transcript 尾部确认代码完整性；(2) 用 `update_implementation_subtask` 将子任务从 `failed` 重置为 `ready`；(3) 重新 claim + dispatch 新 implementer 验证并补 handoff。不要直接强制标记 `done`。

### 4. /model 命令与 Agent Running 互斥
所有模型/thinking 切换路径（`/model <provider>/<modelId>`、`/model thinking <level>`、交互选择）都必须在 `renderer.running` 时拒绝，提示用户先 `/abort`。避免运行中途改模型导致状态不一致。状态同步：`currentModel` 对象包含 `{provider, modelId, displayName, thinkingLevel}`，每次切换后 `updateFrameModel()` 刷新底栏。

### 5. UI 原型门禁审批记录闭环
`awaiting_approval` 阶段 UI 原型获批后，必须同步更新 `ui.md` 和 `plan-review.md` 中的审批状态记录，从"待审批"改为"已批准"并注明时间。Checker 会核验这些文档中的审批记录是否与实际 `approvalGrant` 一致；未更新会导致 checker verdict 为 "Needs work" 阻塞关单。
