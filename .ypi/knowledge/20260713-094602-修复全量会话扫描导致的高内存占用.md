## Summary

修复全量会话扫描导致的高内存占用。根因是 Pi SDK `SessionManager.listAll()` 的 `buildSessionInfo()` 为所有 JSONL 累积完整消息文本并生成 `allMessagesText`。在约 447MB 会话数据、最大约 214MB 单文件的环境中，快照显示 V8 heapUsed 约 941MiB、RSS 约 1125MiB，而 AgentSession/Studio/Terminal/Browser Share 均为空，确认主要是会话扫描堆对象膨胀。

## Reusable knowledge

- 会话列表是元数据场景，不应依赖会保留 `allMessagesText` 的 SDK 全量 inventory。
- 使用 chunk 级 JSON tokenizer/状态机，提取 header、最新 name、messageCount、首条用户文本有界前缀和 activity time；避免 `readFile`、整行 `JSON.parse` 和正文累积。
- active、archive、archive-all、allowed-roots、按 cwd 删除统一使用轻量 scanner；详情、上下文、分支、导出和 Usage 仍可按需读取单个 JSONL 的完整内容。
- 结果对象不得包含 `allMessagesText`；固定并发和单字段上限共同保证内存不随历史正文总量线性增长。
- 兼容性测试需覆盖 JSON escape/chunk 边界、Unicode、content blocks、name clear、Studio child/parent、归档、malformed 文件及大正文内存门禁。
- 已知非阻塞语义差异：归档列表改用首条 user 文本与 activity time；中途损坏 JSONL 比 SDK 更严格，可能整文件不展示。
