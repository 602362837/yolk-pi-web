| `components/SessionSidebar.tsx` | Project Registry-driven project/space tree sidebar. It loads projects from `/api/projects` instead of synthesizing them from sessions, and lazy-loads **active** linked sessions for the selected project space via `GET /api/projects/:projectId/spaces/:spaceId/sessions` (response `{ sessions, legacyUnassigned, studioChildrenByParentSessionId }` only — no `archivedCounts`, no `sessions-archive/` scan). The sidebar does **not** load, expand, or restore archived session lists: there is no “已归档 (N)” block, no `archivedSessions` state, and no client call to `/api/sessions/archived` or unarchive. Empty space state is active-only (`No sessions found in this space`). Clicking the button opens `ProjectSpaceSwitchDialog` for layered (left project list → right space list) browsing, search, and add-project flows inside a viewport-level modal dialog. The `SessionSidebar` owns all project/space state and callbacks (registration from manual path, OS directory picker, Git clone, default directory) and passes them into the dialog as props; the dialog is purely a UI shell that calls back to the sidebar's existing `upsertProjectAndSelectMainSpace()`, `handleDirectoryPicker()`, `commitCustomPath()`, `handleGitCloneSubmit()`, etc. The sidebar header displays current project name / space subtitle / WorkTree badge; WorkTree archive/delete remains available by right-clicking the selected switch button or a WorkTree row inside the dialog. Add-project forms (manual path, Git clone) are rendered inside the dialog with mutually exclusive toggles; empty registry onboarding renders directly in the dialog with large primary buttons for `Add project folder…` / `Add project path…` / `Add project from Git…` / `Use default directory`. Main spaces without a custom display name render as `主空间` while keeping `spaceId: "main"`; the selected space header subtitle shows `空间：...` plus the current Git branch or WorkTree branch/base status, appends path-missing state when needed, and keeps full path/status details in the hover/focus tooltip. WorkTree badges remain compact (`WT` plus branch) with a tooltip for branch and base/unknown state. Session, Studio child, hover-action, and delete-confirmation rows keep title/metadata content on single non-wrapping lines and ellipsize long text at narrow sidebar widths. **WorkTree archive/delete flow:** `confirmWorktreeAction()` optimistically merges the API response's `archivedSpaces` into local project state so the worktree space disappears immediately, then refreshes projects in the background. If the currently-selected space was the archived/deleted one, fallback preference is: (1) `main` space of the same project, (2) any other active non-missing space in that project, (3) API-provided `fallbackCwd`, (4) null. Deleted session ids are forwarded via `onSessionDeleted`. The sidebar **centralizes project and space context menu state** through `ProjectSpaceContextMenuState` (targeting a specific project or project+space pair). Context menu callbacks (`handleDialogProjectContextMenu`, `handleDialogSpaceContextMenu`) bridge dialog right-click events into the sidebar state so that edit/star/archive/WorkTree-delete actions operate on the right-clicked target, not just the currently selected workspace. **Space drag-reorder** is handled by `handleReorderSpaces` which optimistically updates local state and calls `PATCH /api/projects/[projectId]/spaces`, rolling back to pre-drag project state on failure. **Top workspace menu and metadata dialog** labels have been unified to "星标/取消星标" for both projects and spaces; the metadata dialog distinguishes the two with contextual hints ("星标项目优先排序" for projects vs "仅用于标记，不影响空间排序" for spaces). |

## Styles

Global CSS lives in `app/globals.css`. `YpiStudioSessionWidget` keeps the five-station rail semantic state unchanged: only a `current` station backed by active `intake`/`planning`/`implementing`/`checking` receives the presentation-only halo and outgoing shimmer. Waiting, attention/error, terminal, and Review states remain static; expanded-panel dragging pauses rail animation, and reduced-motion uses a static halo/state line. Waiting-approval plan-review UI adds widget action-row button styles (`.ypi-studio-widget-plan-action-row` / `.ypi-studio-widget-plan-review-btn`) and the dedicated modal layer (`.ypi-studio-plan-review-*`: dialog shell, readonly banner, loading/error/empty states, scroll body, mobile near-fullscreen sheet); colors reuse existing `--bg*` / `--border` / `--text*` / `--accent` tokens and honor reduced-motion.

Components may reference these CSS variables directly:

```text
--bg --bg-panel --bg-hover --bg-selected --bg-subtle --border
--text --text-muted --text-dim --accent --accent-hover
--user-bg --assistant-bg --tool-bg --font-mono
```

They are also mapped to Tailwind `--color-*` utility aliases. The theme toggles by adding/removing `dark` on `document.documentElement`.

Session sidebar loading uses stable project/space selection identity, aborts obsolete requests, and ignores `AbortError`. Session list reloads when the selected project/space changes, when the projects registry first resolves that selection (URL `?session=` restore race), when AppShell bumps `refreshKey` (new session / agent end / fork), and when the manual refresh button runs. Reloads only hit the project-space active sessions route; they never request `/api/sessions/archived` and never depend on archive counts. If project/space ids are set before projects load, the sidebar waits instead of clearing the list; it does not depend on full `projects` array identity for every metadata-only project refresh. Known archived session ids can still open read-only chat detail through the session-by-id path (`ChatWindow` archived banner); that is independent of the sidebar list.

### AppShell hydration-safe layout persistence

`components/AppShell.tsx` keeps four localStorage-backed layout values on hydration-safe external stores (`createPersistentLayoutStore` + `useSyncExternalStore`), not `useState(getInitial*)` with unconditional persist effects:

| State | Storage key(s) | Server snapshot | Client notes |
| --- | --- | --- | --- |
| `sidebarWidth` | `pi-web-sidebar-width` | `260` | Clamp 220–520 on read/write. |
| `rightPanelWidth` | `pi-web:right-panel-width` | `300` (`RIGHT_PANEL_MIN_WIDTH`) | Clamp to viewport (min 300, max ~65%); missing/invalid falls back to ~42% viewport after hydration. |
| `explorerHeight` | `pi-web-preview-explorer-height`; legacy `pi-web-sidebar-explorer-height` (one-shot migrate, also in `watchKeys`) | `null` | Min height 120; `null` keeps flex default; write of `null` is a no-op (does not clear stored preference). Legacy migrate runs in client `read` only and must not notify. |
| `explorerOpen` | `pi-web-preview-explorer-open` | `true` | Affects conditional FileExplorer subtree; missing key defaults open (`stored !== "false"`). |

Contract: `getServerSnapshot` is fixed for SSR/hydration first paint; `getSnapshot` returns primitives/`null` with a cached snapshot to avoid render loops; `setValue` normalizes, writes (try/catch), then notifies same-tab subscribers; `storage` listeners are registered only while subscribers exist and filter by primary/watch keys or clear (`key === null`). Hydration may briefly show defaults before correcting to stored preferences; no `suppressHydrationWarning`, no layout blocking script, and no server-side preference storage.
