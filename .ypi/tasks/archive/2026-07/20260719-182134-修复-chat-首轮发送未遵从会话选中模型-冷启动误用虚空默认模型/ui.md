# UI — 无 UI 原型门禁

## 结论

**不触发 UI 原型门禁。不指派 UI 设计员。无需 HTML 原型。**

## 判定依据

| 门禁触发条件 | 本任务是否命中 |
| --- | --- |
| 页面变更 | 否 |
| 前端功能新增（用户可见） | 否 |
| 已有交互结构变化 | 否 |
| 审批/确认体验变化 | 否 |
| 用户可见信息结构变化 | 否 |

本任务是 **Chat 模型 pin 正确性 + 服务端冷启动默认源** 修复：

- 模型选择器组件、下拉/弹窗、文案、布局不变。
- Settings 树与 `yolk.defaultModel` 表单**默认不改交互**。
- 成功路径用户无新提示；失败路径沿用现有 `setError` / 发送失败展示。
- 计划 B 仅改变“后台冷启动落到哪个默认”，用户仍通过既有选择器看到/选择模型。

## 可选文案（非强制 · 不触发门禁）

若实现阶段发现 Settings → 蛋黄𝝅 默认模型说明仍写“仅新建会话”，可**可选**补一句：

> 亦用于 Chat 会话冷启动在无法恢复历史模型时的服务端兜底（不会覆盖会话已选模型；不会写入 Pi settings.json）。

该改动属于静态说明文字，**不改变交互结构**，**不强制** HTML 原型。若产品要求改控件/模式切换，再回退架构师 + UI 设计员。

## 用户可感知行为（无新 UI，仅正确性）

| 场景 | 修复前 | 修复后 |
| --- | --- | --- |
| 历史会话首轮发送 | 选择器显示 M，实际可能跑虚空默认 | 实际跑 M |
| idle 后发送 | 可能再跑虚空默认 | 实际跑 M |
| 无可恢复模型的冷启动中间态 | 可能短暂/实际落到 settings 虚空 | 优先 yolk specific（再被 UI pin 对齐） |
| 选择器展示 | 可已正确 | 保持；不改 priority |

## 审批说明

计划审批时**无需**审 HTML 原型；请审 PRD/Design/Implement/Checks 中的行为契约与计划 B 增量即可。

若实现阶段发现必须新增用户可见错误文案结构或选择器状态机，应 **回退架构师 + UI 设计员** 补原型，不得静默扩大范围。
