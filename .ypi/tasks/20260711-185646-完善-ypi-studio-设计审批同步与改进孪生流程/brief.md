# Brief - 第三版：YPI Studio 改进流程

## 用户要解决的问题

主任务完成实现和检查后，用户在验收时常会说“这里还要调整”。今天这句话没有一个可靠的归宿：它可能只留在聊天里，也可能被错误地变成另一个顶层任务。第三版把它定义为主任务下的**改进流程**：每条用户确认要处理的问题都能被分析、计划、实现、检查和验收，同时主任务始终对最终交付负责。

## 本版的核心路径

1. 主任务到达 `user_acceptance`，main 向用户请求验收。
2. 用户没有问题并明确接受，主任务完成。
3. 用户提出问题后，main 复述问题并获得“创建改进项”的明确确认；每个独立问题创建一个改进项。
4. 主任务改为“等待改进完成”。改进师分析问题，必要时请求 UI 原型和计划批准，随后由实现员、检查员处理。
5. 每个改进项都要由用户验收。全部改进项验收完成后，主任务回到 `review`，main 汇总改动并再次请求主任务验收；不能自动完成。

## 已核实现状

- 一个 Studio 主任务由 `.ypi/tasks/<task-id>/task.json` 和同目录 artifacts 管理，任务列表只扫描顶层任务目录。
- `YpiStudioPanel` 已有任务详情、artifact 预览、成员运行记录和计划审批书；`YpiStudioSessionWidget` 已展示绑定任务摘要。
- Settings 的 Studio 区已有“默认成员”模型/思考强度表，策略读取 `studio.members[member]`，顺序由固定成员数组决定。当前默认成员为 architect、ui-designer、implementer、checker。
- 当前 workflow 没有 user_acceptance、等待改进、改进项存储或 `improver` 默认成员。

## 本版边界

- 设计改进流程、数据和 UI，并保留计划/原型/审批书同步机制。
- 改进项不是顶层任务，也不是普通 child task；它只存活在自己的主任务下面。
- 不自动从任意聊天文本创建改进项，不在 Panel 中提供绕过聊天确认的创建/批准/验收按钮。
- 不修改生产代码、任务状态、implementationPlan、git 提交或推送。

## 仍需确认

1. 是否同意将 `improver` 加为默认 Studio 成员？推荐同意。
2. 失败、取消或用户拒绝某个改进时，是否只有用户明确“接受不处理”才能解除主任务阻塞？推荐同意。
3. 已在原主计划批准范围内的纯修复是否可继承主计划批准？推荐允许，但 UI、范围或风险变化必须单独批准。
