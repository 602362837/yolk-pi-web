# Summary

已完成 YPI Studio 架构师任务拆解与分阶段实现编排：新增结构化 `implementationPlan` / `implementationProgress`，让架构师在 planning 阶段输出可机器读取的子任务计划，父会话在 implementing 内按 next/claim/update 编排，避免实现员一次性承担完整复杂任务。

# Reusable knowledge

- 主状态机保持 `planning -> awaiting_approval -> implementing -> checking`，实现拆解作为内部层存在，不重写工作流。
- `awaiting_approval -> implementing` 仍依赖 approvalGrant 硬门禁；override 不能绕过。子任务 claim/running/done/blocked/skipped 也必须检查主任务已合法进入 `implementing`。
- `implementationPlan` 是静态计划事实源，`implementationProgress` 是运行时状态事实源；旧任务缺失字段时应安全降级为空态。
- `ypi_studio_subagent(subtaskId)` 可把 implementer/checker 限定到单个子任务；派发前应校验 subtaskId 存在于 plan。
- UI 只展示轻量摘要到 task card/widget，完整子任务明细放 Studio detail 的 Implementation 区域，避免长文本压垮浮窗。
- 检查阶段修复：ready/pending 更新只能在 `implementing` 或 `changes_requested`；无效 subtaskId 直接失败。
