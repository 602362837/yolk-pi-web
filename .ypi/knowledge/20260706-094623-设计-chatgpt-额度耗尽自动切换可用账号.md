# Summary

实现并检查通过 ChatGPT/OpenAI Codex 多账号额度耗尽自动切换：默认关闭，通过 Settings → ChatGPT 开关启用；仅 `openai-codex` 明确 quota/usage exhausted 时触发；自动选择可用保存账号、激活、刷新运行中 RPC auth，并对当前失败 turn 受控 retry。任务同时修复了 YPI Studio 当前 session `pi_process_*` runtime pointer 被忽略导致任务绑定/恢复失败的问题。

# Reusable knowledge

- 自动切号必须默认关闭，配置放在 `pi-web.json.chatgpt.autoFailover`，MVP UI 只暴露 enabled。
- 不要把普通 transient `429` / `rate_limit_exceeded` / `too many requests` 当成切号信号；只处理明确 quota/usage exhausted。
- 多 session 并发切号要在 agent run 开始时捕获 `triggerAccountId`，进入进程级 mutex 后重读 active：若 active 已不是 trigger，则返回 `already_switched_by_other_session` 并只 retry，禁止基于新 active 继续 B→C 级联切换；activate 前还要 double-check active。
- 每 turn 维护 attempts/switches budget，默认最多 1 次 retry / 1 次真实切号，避免无限切号和无限 retry。
- 候选账号应排除 active、cooldown、当前 turn 已尝试、认证失败、quota 满额账号；可优先使用新鲜 quota cache，过期时受控调用 quota API。
- 切换 active account 后必须调用 `reloadRpcAuthState()`，让 live wrappers reload auth/model registry，并 cleanup Codex session resources。
- Runtime 接入应先执行 pi 原始 post-run retry/compaction 逻辑；只有原逻辑返回 false 后再执行 failover，避免抢在内置恢复前切号。
- YPI Studio 任务绑定恢复不能无条件忽略 `pi_process_*`；若 runtime pointer 存在，应允许解析，否则当前 session 创建的任务会显示 no_task 并影响 approval/continue。
